# Техническое Задание: Музыкальный Плейлист-Коллаборатор (MusicCollab)

**Версия:** 1.0
**Срок реализации:** 2–3 недели (Sprint)
**Команда:** 2 разработчика (Fullstack ориентированные, но с разделением по слоям/модулям)

## 1. Описание проекта
Веб-приложение для совместного управления музыкальным плейлистом в реальном времени. Пользователи создают «Комнаты», приглашают друзей, добавляют треки (через моки Spotify API) и голосуют за них. Треки в очереди сортируются автоматически на основе рейтинга.

**Основная цель:** Реализовать систему real-time голосования и управления очередью воспроизведения без необходимости интеграции с реальным платным API Spotify на данном этапе.

## 2. Роли пользователей
1.  **Создатель (Host):**
    *   Создает комнату.
    *   Управляет статусом воспроизведения (Play/Pause/Next).
    *   Может удалять любые треки.
2.  **Участник (Guest):**
    *   Присоединяется по коду комнаты.
    *   Может добавлять треки.
    *   Может голосовать (Like/Dislike).
    *   Видит очередь и текущий трек.

## 3. Функциональные требования

### 3.1. Авторизация и Комнаты
*   Вход без пароля (по Nickname).
*   Создание комнаты: генерация уникального 6-значного кода.
*   Присоединение к комнате: ввод кода.
*   Выход из комнаты.
*   *Технически:* Сессия хранится в памяти/Redis, привязка `UserID -> RoomID`.

### 3.2. Поиск и Добавление треков (Mock Spotify)
*   Поиск треков по названию/исполнителю.
*   Бэкенд возвращает мокированные данные (JSON структура, аналогичная Spotify API).
*   Добавление трека в «Очередь» (Queue).
*   Проверка на дубликаты в текущей очереди.

### 3.3. Голосование и Очередь
*   Механика: +1 (Like) / -1 (Dislike) или только +1. *Решение: Score = Likes - Dislikes.*
*   Сортировка очереди:
    1.  Треки с высшим Score.
    2.  При равном Score — по времени добавления (FIFO).
    3.  Текущий воспроизводимый трек всегда первый.
*   Real-time обновление очереди у всех пользователей в комнате при изменении голоса.

### 3.4. Плеер (Визуальный)
*   Отображение текущего трека (Обложка, Название, Артист).
*   Кнопки управления (только у Host): Play, Pause, Skip.
*   Прогресс-бар (симуляция, таймер на клиенте или синхронизированный с сервера).

## 4. Технический стек

*   **Frontend:** React (Vite) + TypeScript. State: Zustand или Redux Toolkit.
*   **Backend:** Node.js (NestJS или Express) + TypeScript.
*   **Real-time:** Socket.io (для синхронизации очереди и плеера).
*   **Database:** PostgreSQL (основные данные) + Redis (опционально для временных сессий комнат, можно хранить в PG для упрощения).
*   **Mock API:** Встроенный сервис на бэкенде, возвращающий статические JSON-данные с задержкой (имитация сети).
*   **Контейнеризация:** Docker & Docker Compose.

## 5. Архитектура и Модель Данных

### 5.1. Схема БД (PostgreSQL)
*   `Users`: `id`, `nickname`, `avatar_url`.
*   `Rooms`: `id`, `code` (unique), `host_id`, `created_at`, `is_active`.
*   `Tracks`: `id`, `spotify_mock_id`, `title`, `artist`, `cover_url`, `duration`.
*   `Room_Tracks` (Queue): `id`, `room_id`, `track_id`, `added_by_user_id`, `score` (default 0), `position`, `status` (queued, playing, played).
*   `Votes`: `id`, `room_track_id`, `user_id`, `value` (+1/-1).

### 5.2. API Endpoints (REST)
*   `POST /auth/login` — вход/регистрация.
*   `POST /rooms` — создать комнату.
*   `POST /rooms/:code/join` — войти в комнату.
*   `GET /tracks/search?q=` — поиск (Mock).
*   `POST /rooms/:id/tracks` — добавить трек в очередь.

### 5.3. WebSocket Events (Socket.io)
*   `join_room` / `leave_room`.
*   `queue_update` — сервер рассылает всем обновленный список очереди.
*   `vote_track` — клиент отправляет голос, сервер пересчитывает и шлет `queue_update`.
*   `player_state` — изменение статуса плеера (Host меняет -> все видят).

## 6. Распределение задач (На 2 человек)

Для эффективной работы предлагается разделение по **слоям с пересечением на контрактах**.

### Разработчик 1 (Backend & Core Logic Lead)
**Фокус:** Сервер, БД, WebSocket логика, Алгоритмы, Mock API.

1.  **Настройка проекта:**
    *   Инициализация репозитория, Docker Compose (DB + Backend).
    *   Настройка TypeScript, линтеров.
2.  **База данных:**
    *   Проектирование схемы БД (миграции).
    *   Репозитории/ORM (TypeORM или Prisma).
3.  **API & Mock:**
    *   Реализация REST endpoints (Auth, Rooms, Search).
    *   Написание сервиса Mock Spotify (генерация фейковых данных, поиск по массиву).
4.  **WebSocket Logic:**
    *   Управление комнатами в памяти (или Redis).
    *   Логика пересчета очереди (Sorting Algorithm).
    *   Обработка событий голосования и синхронизация состояния.
5.  **Безопасность:**
    *   Валидация прав (только Host может скипнуть трек).

### Разработчик 2 (Frontend & Integration Lead)
**Фокус:** UI/UX, Клиентская логика, WebSocket Client, Интеграция.

1.  **Настройка проекта:**
    *   Инициализация React приложения, настройка роутинга.
    *   Настройка UI-кита (Material UI, Chakra или Tailwind).
2.  **State Management:**
    *   Глобальный стейт (пользователь, текущая комната, сокет-коннекшн).
    *   Кэширование запросов (React Query / SWR).
3.  **Компоненты:**
    *   Экран входа / Создания комнаты.
    *   Компонент Плеера (визуал, прогресс-бар).
    *   Список очереди (Drag-and-drop не нужен, но нужна красивая сортировка).
    *   Карточка трека (кнопки голосования).
4.  **Интеграция:**
    *   Подключение к Socket.io клиенту.
    *   Обработка событий `queue_update` (оптимистичный UI при голосовании).
    *   Синхронизация REST запросов и WS событий.
5.  **Адаптивность:**
    *   Мобильная версия (так как часто используют с телефона).

### Совместные задачи (Обязательно)
1.  **День 1:** Согласование API контрактов (Swagger/OpenAPI) и схемы БД.
2.  **День 3:** Интеграция Frontend <-> Backend (проверка авторизации и создания комнаты).
3.  **День 7:** Тестирование WebSocket синхронизации (открыть 2 браузера, проверить обновление в реальном времени).
4.  **Финал:** Code Review друг друга, деплой.

## 7. Критерии приемки (Definition of Done)

1.  **Функциональность:**
    *   Можно создать комнату и зайти в нее с другого устройства.
    *   Поиск возвращает список треков.
    *   Добавленный трек появляется у всех участников без перезагрузки страницы.
    *   Голос меняет позицию трека в очереди у всех пользователей мгновенно.
    *   Host может переключить трек, у всех обновится экран «Now Playing».
2.  **Код:**
    *   Отсутствуют `console.log` в продакшене.
    *   Код типизирован (TypeScript strict mode).
    *   Есть базовая обработка ошибок (try/catch, UI уведомления).
3.  **Документация:**
    *   README с инструкцией по запуску (`docker-compose up`).
    *   Список переменных окружения (`.env.example`).

## 8. Риски и План Б

| Риск | Вероятность | Митигация (План Б) |
| :--- | :--- | :--- |
| Сложности с WebSocket синхронизацией | Высокая | Использовать Polling (опрос сервера каждые 2 сек) как запасной вариант, если WS не успеют. |
| Mock API слишком скучный | Средняя | Добавить рандомизацию в обложки и длительность треков для живости. |
| Конфликты Git | Средняя | Четкое разделение папок: `/server` и `/client`. Веточная модель: `feat/backend-xxx`, `feat/frontend-yyy`. |
| Не хватает времени на дизайн | Высокая | Использовать готовую UI библиотеку, не писать кастомные стили с нуля. |

## 9. Таймлайн (Пример на 2 недели)

*   **День 1-2:** Сетап, БД, API Контракт, Базовая верстка.
*   **День 3-5:** Реализация Комнат и Поиска треков (REST).
*   **День 6-8:** Реализация WebSocket, Голосования и Сортировки очереди.
*   **День 9-10:** Плеер, права доступа (Host/Guest), полировка UI.
*   **День 11-12:** Тестирование, багфикс, подготовка к демо.

### Приложение Б: Логика сортировки очереди
```typescript
function sortQueue(tracks: Track[]) {
  return tracks.sort((a, b) => {
    if (a.status === 'playing') return -1;
    if (b.status === 'playing') return 1;
    if (b.score !== a.score) return b.score - a.score;
    return a.addedAt - b.addedAt;
  });
}
```
