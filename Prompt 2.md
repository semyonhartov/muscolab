Давай уточню, вход без авторизации означает, что пользователь авторизуется только через Spotify?

---

Нет, в предыдущем варианте под «входом без авторизации» подразумевался **упрощенный вход по никнейму** (без пароля и без сторонних сервисов), чтобы сосредоточиться на логике очередей и голосования, не усложняя проект настройкой OAuth.

Однако, для проекта уровня «Spotify Collaborator» **авторизация через Spotify (OAuth 2.0)** является более правильным и профессиональным решением. Это даст реальные идентификаторы пользователей, аватарки и возможность в будущем легко переключиться на реальный API Spotify для воспроизведения.

Ниже представлена **обновленная версия Технического Задания**, где авторизация заменена на **Spotify OAuth**, а архитектура адаптирована под это изменение.

# Обновленное Техническое Задание: Музыкальный Плейлист-Коллаборатор (v2.0)

**Версия:** 2.0 (С учетом Spotify OAuth)
**Команда:** 2 разработчика
**Стек:** React, Node.js, Socket.io, PostgreSQL, Spotify API (Mock или Real)

## 1. Изменения в концепции авторизации
Вместо ввода никнейма реализуется полноценный вход через **Spotify OAuth 2.0**.
*   **Плюсы:** Реальные профили пользователей, аватарки, уникальные ID, готовность к интеграции с реальным плеером Spotify.
*   **Минусы:** Требуется регистрация приложения в Spotify Developer Dashboard, настройка redirect URIs.

## 2. Функциональные требования (Обновленные)

### 2.1. Авторизация (Spotify OAuth)
*   **Кнопка «Login with Spotify»** на главном экране.
*   **Flow:** Authorization Code Flow (рекомендуется для безопасности) или Implicit Grant (проще для SPA, но менее безопасно).
*   **Токены:** Бэкенд получает `access_token` (для запросов к API) и `refresh_token` (для обновления сессии).
*   **Профиль:** После входа система забирает `display_name`, `id`, `images` из профиля Spotify и сохраняет локальную сессию.

### 2.2. Комнаты и Участники
*   Идентификатор пользователя в комнате — это `spotify_user_id`.
*   Отображение аватарки пользователя из Spotify рядом с добавленными треками.
*   Host комнаты — тот, кто создал её (первый вошел).

### 2.3. Поиск треков (Mock vs Real)
*   **Вариант А (Mock):** Бэкенд игнорирует токен Spotify при поиске и возвращает фейковые данные (как в ТЗ v1.0). *Выбрано как основное требование.*
*   **Вариант Б (Real):** Бэкенд использует `access_token` пользователя для поиска через реальный Spotify API `/v1/search`.
*   *Рекомендация для команды:* Реализовать интерфейс так, чтобы можно было переключить источник данных (Mock/Real) через переменную окружения.

### 2.4. Голосование и Очередь
*   Без изменений относительно v1.0.
*   Важно: Запрет на повторное голосование одним и тем же `spotify_user_id` за один трек.

## 3. Техническая реализация (Обновленная)

### 3.1. Настройка Spotify Developer
1.  Зарегистрировать приложение на [Spotify Developer Dashboard](https://developer.spotify.com/dashboard).
2.  Получить `Client ID` и `Client Secret`.
3.  Указать `Redirect URI` (например, `http://localhost:3000/auth/callback` для локальной разработки).

### 3.2. Backend (Node.js)
*   **Библиотеки:** `passport-spotify` (для Express/NestJS) или ручная реализация OAuth flow.
*   **Endpoints:**
    *   `GET /auth/spotify` — редирект на страницу логина Spotify.
    *   `GET /auth/callback` — обработка кода, обмен на токен, создание сессии.
    *   `GET /auth/me` — получение данных текущего пользователя.
    *   `GET /logout` — очистка сессии.
*   **Хранение токенов:** В продакшене токены нужно шифровать в БД. Для MVP можно хранить в памяти/Redis.

### 3.3. Frontend (React)
*   **Библиотеки:** `react-spotify-web-playback` (опционально, если будет реальный плеер), или кастомная реализация OAuth редиректа.
*   **Роутинг:** Защита приватных роутов (если нет токена — редирект на логин).
*   **UI:** Отображение аватарки пользователя в хедере.

## 4. Распределение задач (Обновленное)

### Разработчик 1 (Backend & Auth Lead)
1.  **Spotify App Setup:** Регистрация приложения, получение кредов.
2.  **Auth Logic:** Реализация OAuth flow на бэкенде (обмен кода на токен).
3.  **Session Management:** Привязка сессии к `spotify_user_id`.
4.  **Mock API Service:** Обновление сервиса моков, чтобы они возвращали данные, совместимые с реальным ответом Spotify (для бесшовной замены).
5.  **WebSocket Security:** Валидация токена пользователя при подключении к сокету.

### Разработчик 2 (Frontend & UX Lead)
1.  **Login Page:** Верстка страницы с кнопкой «Login with Spotify» (использовать официальные гайдлайны Spotify Branding).
2.  **Auth State:** Обработка редиректа после логина, сохранение токена/сессии.
3.  **User Profile:** Отображение аватарки и имени в интерфейсе комнаты.
4.  **Error Handling:** Обработка случаев, когда пользователь отменяет вход в Spotify.
5.  **Integration:** Связка фронтенда с новыми эндпоинтами авторизации.

## 5. Риски и решения (Специфичные для OAuth)

| Риск | Решение |
| :--- | :--- |
| **Истекание токена** | Access token живет 1 час. Реализовать логику Refresh Token на бэкенде для продления сессии. |
| **CORS ошибки** | Настроить правильные CORS политики на бэкенде для домена Spotify и локалхоста. |
| **Сложность тестирования** | Создать тестовых пользователей Spotify для команды, чтобы не блокировать аккаунты друг друга. |
| **Mock vs Real** | Четко разделить слой `MusicProvider`. Интерфейс должен быть одинаковым для Mock и Real API. |
